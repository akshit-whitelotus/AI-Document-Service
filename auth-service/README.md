# auth-service

A standalone authentication microservice. It owns its own PostgreSQL
database and is the only service that ever touches passwords. Other
services (like `ai-document-processing-service`) never see credentials —
they only ever see a validated identity, extracted from a JWT they verify
themselves using this service's published public key.

## Architecture

**Token strategy: short-lived JWT access tokens (RS256) + rotating, DB-backed refresh tokens.**

- **Access tokens** are stateless JWTs, signed with this service's RSA
  private key, valid for 15 minutes by default. Any service (e.g. the
  document service) can verify them **locally**, without calling this
  service on every request, by fetching this service's public key from
  `/auth/.well-known/jwks.json` and caching it.
- **Refresh tokens** are opaque random strings. Only their SHA-256 hash is
  ever stored in the database — a database leak alone can't be replayed
  into a session. Every refresh **rotates**: using a refresh token
  immediately revokes it and issues a new one. If a revoked refresh token
  is ever reused, that's a strong signal of theft, and the request is
  rejected.

**Trade-off being made:** because access tokens are verified locally and
statelessly, revoking a user's access (e.g. an admin disabling an account)
doesn't take effect until their current access token expires (≤15 minutes
by default). Refresh tokens, by contrast, are revoked instantly (checked
against the DB on every refresh). If your use case needs *instant*
access-token revocation, the alternative is opaque access tokens validated
via an introspection endpoint on every request — trading latency/coupling
for immediate revocation. This service does not implement that; it
optimizes for lower latency and lower coupling between services.

## Database schema

- **`users`** — `id` (UUID, not sequential — user IDs end up in JWTs
  and get passed to other services), `email` (unique, indexed),
  `hashed_password` (argon2id), `is_active`, timestamps.
- **`refresh_tokens`** — `token_hash` (SHA-256 of the raw token, unique,
  indexed), `user_id` (FK, indexed), `expires_at`, `revoked_at`
  (nullable — NULL means still valid), plus `user_agent`/`ip_address`
  for audit.
- **`login_history`** — audit trail of every login attempt, success or
  failure. `user_id` is nullable because a failed attempt against a
  non-existent email still needs to be recorded, with no user to link to.

Migrations are managed with Alembic (`alembic/`), run over a **sync**
Postgres connection (`psycopg`) even though the running app uses an
**async** connection (`asyncpg`) — Alembic's migration runner is
synchronous, so this is the standard pattern for an async SQLAlchemy app.

## Endpoints

| Method | Path | Description |
|---|---|---|
| POST | `/auth/register` | Create a new user. Rate-limited. |
| POST | `/auth/login` | Exchange email/password for an access + refresh token pair. Rate-limited. |
| POST | `/auth/refresh` | Exchange a valid refresh token for a new access + refresh token pair (rotation). |
| POST | `/auth/logout` | Revoke a refresh token. |
| GET | `/auth/me` | Return the current user, given a valid `Authorization: Bearer <access_token>` header. |
| GET | `/auth/.well-known/jwks.json` | Public key set for verifying access tokens locally. |
| GET | `/healthz` | Liveness check. |

Both `/auth/login` and `/auth/register` return **identical** error
responses for "email doesn't exist" and "wrong password" — this prevents
using the error message to enumerate registered emails.

## Local setup (no Docker)

1. **Postgres.** Make sure PostgreSQL is running locally and create a
   database for this service:
   ```bash
   createdb auth_service_db
   ```

2. **Virtualenv + dependencies:**
   ```bash
   cd auth-service
   python3 -m venv .venv
   source .venv/bin/activate
   pip install -r requirements.txt
   ```

3. **Configure:**
   ```bash
   cp .env.example .env
   # edit .env if your Postgres isn't on localhost:5432 with postgres/postgres
   ```

4. **Run migrations:**
   ```bash
   alembic upgrade head
   ```
   This also generates an RSA keypair on first run (`keys/private_key.pem`,
   `keys/public_key.pem`) if one doesn't already exist — **do not commit
   these to version control.**

5. **Start the service:**
   ```bash
   uvicorn app.main:app --reload --port 8001
   ```

6. **Run tests** (uses an in-memory SQLite DB, never touches your real
   Postgres database):
   ```bash
   pytest -v
   ```

## Creating a new migration

After changing a model in `app/models/`:
```bash
alembic revision --autogenerate -m "describe the change"
alembic upgrade head
```
Always read the autogenerated migration before applying it — autogenerate
is a good first draft, not a guarantee.

## Integrating with another service

Any service that wants to trust this service's tokens should:

1. Fetch `GET /auth/.well-known/jwks.json` and cache the response (refresh
   the cache periodically, e.g. every few hours, or on a signature
   verification failure in case the key rotated).
2. Verify incoming JWTs locally against the cached public key:
   - Algorithm: `RS256`
   - Issuer (`iss`): value of `JWT_ISSUER` (default `auth-service`)
   - Audience (`aud`): value of `JWT_AUDIENCE` — the consuming service
     should set this to its own name and this service should be
     configured to issue tokens with that same audience
3. Trust the `sub` claim as the authenticated user's ID — nothing else
   needs to be shared between the services, and this service is never
   called on a per-request basis for this to work.

See `ai-document-processing-service`'s `app/core/auth.py` and
`app/middleware/` for a concrete example of this integration.
